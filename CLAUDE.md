# CLAUDE.md - spec-driven-todo-ai

## 1. Project Overview

**Project Name**: spec-driven-todo-ai
**Phase**: Phase I - In-Memory Python Console App
**Purpose**: A command-line todo application utilizing spec-driven development methodology where specifications drive implementation through Claude Code generation rather than manual coding.
**Development Approach**: Specifications → Claude Code generates implementation → Testing
**Hackathon**: Hackathon II (Panaversity)

This project follows a spec-driven development approach where all implementation is generated by Claude Code based on detailed specifications. The application is a console-based todo list manager with CRUD operations for tasks, implemented in Python with in-memory storage.

## 2. Technology Stack

- **Python**: Version 3.13+ (required)
- **Virtual Environment**: UV (required for all project operations)
- **Package Manager**: UV (used for dependency management)
- **Development Environment**: Claude Code CLI
- **Standard Library**: Phase I uses only Python standard library (no external dependencies)
- **Architecture**: Console application with MVC-like separation

## 4. Coding Conventions

- **PEP 8 Compliance**: Follow all PEP 8 guidelines strictly
- **Type Hints**: Include type hints on all functions, methods, and variables
- **Docstrings**: Include comprehensive docstrings for all functions, classes, and modules
- **Variable Naming**: Use descriptive, meaningful variable names (avoid single letters except i, j in loops)
- **Line Length**: Maximum 88 characters per line
- **Error Handling**: Implement robust error handling for all user inputs and edge cases
- **Phase I Constraint**: Use only Python standard library (no external dependencies)
- **Timestamp Handling**: Use `datetime.now()` for created_at and updated_at fields
- **User Experience**: Display friendly error messages to users

## 6. Important Notes

- **Spec-Driven Development**: Never write code manually without a corresponding specification in the specs/ directory
- **Claude Code Requirement**: All implementation must be generated via Claude Code based on specifications - no manual coding allowed in Phase I
- **Testing Compliance**: All implementations must satisfy acceptance criteria defined in specs/features/task-crud.md
- **No Database**: Phase I utilizes in-memory storage only (using Python dict for persistence)
- **Timestamp Implementation**: Use `datetime.now()` for both created_at and updated_at fields in Task objects
- **Input Validation**: All user inputs must be validated with appropriate error handling and user-friendly error messages
- **Specification Priority**: When in doubt, refer to specifications in specs/ directory as the source of truth for implementation requirements
